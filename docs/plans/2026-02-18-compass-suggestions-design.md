# Compass提案機能 設計書

## 概要

判断結果表示時に、Compassに登録された夢・目標から「今日やれる具体的なアクション」をカード型で提案する。ユーザーは通常の判断結果を採用するか、Compass提案を採用するか選べる。

**目的**: Compassを「受動的な参照情報」から「能動的な提案エンジン」に進化させる。ユーザーがタスクとして入力し忘れていても、登録した夢から「それよりこっちやった方がいいよ」と提案できるようにする。

## 全体フロー

```
ユーザーがタスク入力 → 時間・エネルギー選択 → 「判断する」
                                                    │
                     ┌──────────────────────────────┤
                     ▼                              ▼
              通常の判断処理                  Compass提案生成（並列）
              （既存のまま）                        │
                     │                    ①放置された夢を検出（ベクトル）
                     │                    ②具体アクションを生成（LLM）
                     │                              │
                     ▼                              ▼
              ┌─────────────────────────────────────────┐
              │  判断結果画面                             │
              │                                         │
              │  【通常の判断結果】（既存のまま）           │
              │  「Aを最優先でやりましょう」               │
              │                                         │
              │  ─────────────────────                  │
              │                                         │
              │  【Compassからの提案】カード型             │
              │  「英語で海外で働く」に最近取り組めて       │
              │  いません。今日30分あるなら：              │
              │  「英語の技術記事を1つ読んで要約する」      │
              │          [このタスクを追加して再判断]       │
              └─────────────────────────────────────────┘
                              │
                     タップした場合
                              ▼
                タスクリストに追加 → 自動で再判断実行
```

## 設計詳細

### 1. 放置された夢の検出ロジック

#### Phase 1（MVP — 履歴なし）

1. ユーザーの全Compassアイテムを取得
2. 今日入力されたタスクと各Compassアイテムのベクトル類似度を計算
3. **入力タスクとの類似度が最も低いCompassアイテム** = 今日忘れている夢
4. そのCompassアイテム + 今日の時間・エネルギーをLLMに渡して具体アクションを生成

#### Phase 2（履歴あり）

Phase 1に加えて：

1. 直近14日間の `TaskDecision` から過去の実行タスクを取得
2. 各タスク名をベクトル化（判断保存時に事前計算してDBに格納）
3. 各Compassアイテムに対して「直近14日間で最も類似度が高かったタスクのスコア」を計算
4. **最近どのタスクとも似ていないCompassアイテム** = 長期間放置された夢

放置スコア計算式:

```
放置スコア = (1 - 今日のタスクとの最大類似度) × 0.4
           + (1 - 過去14日の最大類似度)       × 0.6
```

過去の放置期間を重めに見る（0.6）ことで、長く忘れている夢ほど優先的に提案される。

#### Phase 2 用テーブル: TaskEmbedding

```
TaskEmbedding
  id
  taskDecisionId → TaskDecision
  taskName       (タスク名テキスト)
  embedding      (Bytes - float32)
  createdAt
```

### 2. 具体アクション生成（LLMプロンプト）

検出された「放置されている夢」+ 今日のコンテキストをLLMに渡して具体アクションを生成。

プロンプト: `prompts/compass/suggest-action.md`

ルール:
- ユーザーの利用可能時間内で完了できるアクションにすること
- エネルギーレベルに合った負荷にすること（1-2: 受動的、3: 中程度、4-5: 能動的）
- 曖昧な提案禁止（×「英語を勉強する」→ ○「英語の技術記事を1つ読んで、知らなかった単語を3つメモする」）
- 提案は1つだけ（選択肢が多いと行動できない）
- JSON形式で出力

レスポンス形式:

```json
{
  "compassItemId": "xxx",
  "compassTitle": "英語で海外で働きたい",
  "suggestedTask": "英語の技術記事を1つ読んで、知らなかった単語を3つメモする",
  "reason": "『英語で海外で働く』という夢に最近14日間取り組めていません。今日30分とエネルギー3なら、軽めのインプットから始めるのがおすすめです。",
  "timeEstimate": 25
}
```

### 3. API設計

#### `POST /api/compass/suggest`

通常の `/api/decide` とは別エンドポイント。

```typescript
// Request
{
  tasks: string[],         // 今日入力したタスク
  timeMinutes: number,     // 利用可能時間
  energyLevel: number      // エネルギー 1-5
}

// Response
{
  suggestion: {
    compassItemId: string,
    compassTitle: string,
    suggestedTask: string,
    reason: string,
    timeEstimate: number
  } | null               // Compassが空なら null
}
```

別エンドポイントにする理由:
- 並列呼び出しが簡単（フロントから同時にfetch）
- 通常判断のストリーミングに影響しない
- Compass提案だけ失敗しても通常判断は表示できる

### 4. UIコンポーネント

#### `CompassSuggestionCard`

```
┌─────────────────────────────────────────────┐
│ 🧭 Compassからの提案                         │
│                                             │
│ 「英語で海外で働く」に最近取り組めていません      │
│                                             │
│ 💡 英語の技術記事を1つ読んで、                 │
│    知らなかった単語を3つメモする（約25分）       │
│                                             │
│ 今日30分とエネルギー3なら、軽めの              │
│ インプットから始めるのがおすすめです。            │
│                                             │
│         [ このタスクを追加して再判断 ]          │
└─────────────────────────────────────────────┘
```

- 通常判断のストリーミング完了後にフェードインで表示
- ボタンタップ → タスクリストに追加 → 自動で再判断実行
- 既存の `DecisionResult.tsx` の下に配置

### 5. 再判断フロー

1. 提案タスクを既存タスクリストに追加
2. 同じ時間・エネルギー設定のまま `POST /api/decide` を再実行
3. UIは `loading` → `result` のステップに遷移（既存フロー再利用）
4. 新しい結果にも再びCompass提案が付く（別の夢が提案される可能性あり）

```
src/components/
  CompassSuggestionCard.tsx ← UIカード

prompts/compass/

```

## 実装フェーズ

### Phase 1（MVP — 履歴なし）

1. `neglect-detector.ts` — 今日のタスクとCompassの類似度比較で最も遠い夢を検出
2. `suggest-action.md` — プロンプト作成
3. `compass-suggester.ts` — 検出 → LLMアクション生成の統合
4. `POST /api/compass/suggest` — エンドポイント
5. `CompassSuggestionCard.tsx` — UIカード
6. `ChatDashboard.tsx` — 並列fetch + 再判断フロー接続

### Phase 2（履歴対応）

7. `TaskEmbedding` テーブル追加（Prismaマイグレーション）
8. `/api/decide` の保存処理でタスク名のembeddingも保存
9. `neglect-detector.ts` に過去14日間の履歴分析を追加
10. 放置スコアの重み付け計算（今日0.4 + 過去0.6）
